HTML Reference
The structure of every web page. Master HTML and you control the skeleton of the web.

Document Structure

<!-- Standard HTML5 Boilerplate -->
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <meta name="description" content="Page description here">
  <title>Page Title</title>
  <link rel="stylesheet" href="style.css">
</head>
<body>
  <!-- Content here -->
  <script src="script.js" defer></script>
</body>
</html>

Semantic Elements

ElementPurposeWhen to use
<header>Intro content, logos, navTop of page or section
<nav>Navigation linksMain or secondary navigation
<main>Primary page contentOnce per page
<section>Thematic groupingRelated content with a heading
<article>Self-contained contentBlog posts, news items, cards
<aside>Tangential contentSidebars, related links
<footer>Closing contentCopyright, links, credits
<figure>Media with captionImages, diagrams, code blocks

Links, Images & Forms

<!-- Link (always use noopener for external) -->
<a href="https://example.com" target="_blank" rel="noopener noreferrer">Text</a>

<!-- Image with lazy loading -->
<img src="image.jpg" alt="Descriptive text" width="400" height="300" loading="lazy">

<!-- Accessible form -->
<form action="/submit" method="POST">
  <label for="email">Email</label>
  <input type="email" id="email" name="email" required>
  <button type="submit">Submit</button>
</form>

Common Mistakes

  • Using <div> for everything — use semantic elements instead
  • Missing alt text on images — breaks accessibility and SEO
  • Opening external links without rel="noopener noreferrer"
  • Multiple <h1> tags on one page — only one per page
  • Forgetting the viewport meta tag — breaks all mobile layouts
  • Not associating labels with inputs — breaks screen readers

Best Practices Checklist

  • Include DOCTYPE and lang attribute
  • Add meaningful alt text to every image
  • Use semantic HTML over generic divs
  • Associate every label with its input via for/id
  • Load scripts with defer attribute
  • Add loading="lazy" to below-the-fold images

Flexbox
One-dimensional layout. Perfect for navbars, cards, and aligning items in a row or column.

Container Properties

.container {
  display: flex;
  flex-direction: row;          /* row | row-reverse | column | column-reverse */
  flex-wrap: wrap;              /* nowrap | wrap | wrap-reverse */
  justify-content: space-between; /* main axis */
  align-items: center;          /* cross axis */
  gap: 16px;
}

justify-content Values

ValueEffect
flex-startItems at start (default)
flex-endItems at end
centerItems centered
space-betweenFirst/last at edges, equal space between
space-aroundEqual space around each item
space-evenlyEqual space everywhere including edges

Item Properties

.item {
  flex: 1;             /* grow shrink basis shorthand */
  flex-grow: 1;        /* how much to grow */
  flex-shrink: 0;      /* 0 = don't shrink */
  flex-basis: 200px;   /* starting size */
  align-self: center;  /* override align-items for this item */
  order: -1;           /* visual reorder */
}

Common Patterns

/* Perfect center */
.center { display: flex; justify-content: center; align-items: center; }

/* Navbar */
nav { display: flex; justify-content: space-between; align-items: center; }

/* Card grid that wraps */
.cards { display: flex; flex-wrap: wrap; gap: 16px; }
.card  { flex: 1 1 280px; } /* min 280px, grows to fill */

/* Push last item to far right */
.row .last-item { margin-left: auto; }
💡
Memory trick: justify-content = main axis (same direction as flex-direction). align-items = cross axis (perpendicular to flex-direction).

CSS Grid
Two-dimensional layout. The most powerful layout tool on the web.

Container Setup

.grid {
  display: grid;
  grid-template-columns: repeat(3, 1fr);    /* 3 equal columns */
  grid-template-rows: auto 1fr auto;       /* header, main, footer */
  gap: 24px;                               /* row and column gap */
  column-gap: 16px; row-gap: 24px;        /* or separate */
}

Key Functions & Units

SyntaxMeaning
frFraction of remaining space
repeat(3, 1fr)3 equal columns
minmax(200px, 1fr)Min 200px, max fills remaining
auto-fillAs many columns as fit
auto-fitSame but collapses empty tracks

Placing Items & Named Areas

.item { grid-column: 1 / 3; grid-row: 2 / 4; }  /* by line numbers */
.wide { grid-column: span 2; }                     /* span 2 tracks */

.layout {
  grid-template-areas:
    "header header"
    "sidebar main"
    "footer footer";
}
.header  { grid-area: header; }
.sidebar { grid-area: sidebar; }
.main    { grid-area: main; }
.footer  { grid-area: footer; }

Responsive — No Media Queries Needed

.cards {
  display: grid;
  grid-template-columns: repeat(auto-fill, minmax(260px, 1fr));
  gap: 16px;
}
/* Cards are ≥260px wide, wrap automatically */
💡
Grid vs Flexbox: Use Grid for 2D (rows AND columns). Use Flexbox for 1D — aligning items in a single row or column.

Express.js
The most popular Node.js framework. Build REST APIs fast with minimal setup.

Server Setup & Routes

const express = require('express');
const app = express();

app.use(express.json());
app.use(express.static('public'));

app.get('/users', (req, res) => res.json({ users: [] }));

app.post('/users', (req, res) => {
  const { name, email } = req.body;
  res.status(201).json({ message: 'Created', name });
});

app.put('/users/:id', (req, res) => {
  const { id } = req.params;
  res.json({ updated: id });
});

app.delete('/users/:id', (req, res) => res.status(204).send());

app.listen(3000, () => console.log('Server on 3000'));

Router & Error Handling

// routes/users.js
const router = require('express').Router();
router.get('/', getUsers);
router.post('/', createUser);
module.exports = router;

// app.js — mount routes
app.use('/api/users', require('./routes/users'));

// Error middleware (4 params — required)
app.use((err, req, res, next) => {
  res.status(err.status || 500).json({ error: err.message });
});

Regular Expressions

Character Classes & Quantifiers

PatternMatches

Accessibility Checklist
Build for everyone. Accessibility is not optional — it's core quality.

Structure & Images

  • Page has a single <h1> — the page title
  • Headings are in logical order (h1 → h2 → h3)
  • Semantic HTML used (nav, main, header, footer)
  • lang attribute set on <html> element
  • All images have meaningful alt text
  • Decorative images have alt="" (empty)

Forms & Keyboard

  • Every input has an associated <label>
  • Required fields are clearly marked
  • Error messages explain how to fix the problem
  • All interactive elements are focusable with Tab
  • Focus indicator is visible (never outline: none!)
  • Modals trap focus and close with Escape key

Color & Contrast

  • Text contrast ratio ≥ 4.5:1 (normal), ≥ 3:1 (large)
  • Information not conveyed by color alone
  • Links are distinguishable from body text
💡
Test your site with keyboard only. If you can't navigate using Tab, arrow keys, and Enter — screen reader users can't either.

Security Best Practices
Security is not a feature — it's a foundation. Build it in from the start.

Authentication

  • Passwords hashed with bcrypt (never stored plain)
  • JWT tokens have expiry (short-lived access tokens)
  • Rate limiting on login endpoints
  • HTTPS only — never tokens over HTTP

Input & API Security

  • Validate all user input on the SERVER
  • Sanitize input before storing or displaying
  • Use parameterized queries — never string-concatenate SQL
  • API keys in environment variables only
  • CORS configured to allow only trusted origins
  • Error messages don't leak stack traces
🚨 Never do this:
  • Store passwords in plain text
  • Commit API keys or .env to git
  • Use innerHTML with unescaped user input (XSS)
  • Trust client-side data for authorization
  • Use eval() with user-supplied strings

Developer Glossary
Know the words developers use. Understand any conversation, doc, or job description.
API
Application Programming Interface. A way for two programs to talk to each other. Think of it as a waiter — it carries your request to the kitchen (server) and brings back your food (data).
CLI
Command Line Interface. A text-based way to interact with your computer — the terminal window where you type commands like git, npm, node.
CRUD
Create, Read, Update, Delete — the four fundamental operations of any data-driven application.
DOM
Document Object Model. The tree representation of your HTML that JavaScript can read and modify.
Endpoint
A specific URL in an API. Example: /api/users is an endpoint that returns a list of users when you make a GET request.
Framework
A pre-built set of tools and conventions for building apps faster. Unlike a library, a framework dictates the structure. Examples: React, Vue, Express.
Git
Version control system. Tracks every change you make to code so you can go back in time and collaborate with others.
JWT
JSON Web Token. A compact, signed token used to verify identity between client and server without storing sessions.
Middleware
Functions that run between receiving a request and sending a response in a server. Used for logging, authentication, and parsing.
MVP
Minimum Viable Product. The simplest version of your product that solves the core problem. Build this first, then iterate.
PWA
Progressive Web App. A website that works like a native mobile app — installable on the home screen, works offline.
Refactoring
Improving code structure without changing what it does. Cleaner, faster, more maintainable — same behavior.
REST
Representational State Transfer. An architectural style for APIs using HTTP methods and URLs to represent resources.
SPA
Single Page Application. A web app that loads once and updates content dynamically — no full page reloads. React apps are usually SPAs.
Version Control
A system that records changes to files over time. Git is the industry-standard version control system.

Interview Preparation Guide
These are the questions and concepts that actually come up. Prepare smart.

HTML / CSS Questions

What's the difference between <div> and <span>? ▼
<div> is block-level — it starts a new line and takes full width. <span> is inline — it flows within text. Use div for layout sections, span for styling parts of text inline.
Explain the CSS box model. ▼
Every element is a rectangular box: content → padding → border → margin. With box-sizing: border-box, padding and border are included inside the declared width — making layout predictable.
What is CSS specificity? ▼
Specificity determines which CSS rule wins. Order from highest: !important → inline styles → IDs (100 pts) → classes (10 pts) → elements (1 pt). Higher score wins. When equal, the last rule wins.

JavaScript Questions

What's the difference between == and ===? ▼
== compares with type coercion — it converts types to match before comparing (0 == false is true). === compares value AND type without conversion (0 === false is false). Always use === in real code.
What is a closure? ▼
A closure is when a function "remembers" variables from its outer scope even after that outer function has finished running. Every function in JavaScript forms a closure. Common use: counters, data privacy.
What is the event loop? ▼
JavaScript is single-threaded but non-blocking. The event loop checks if the call stack is empty, then moves callbacks from the queue into the stack. This allows async code (setTimeout, fetch) to work without freezing the page.
Explain async/await. ▼
async/await is syntax for working with Promises. async makes a function return a Promise. await pauses execution inside that function until the Promise resolves — making async code read like synchronous code. Always use try/catch for error handling.

Behavioural Questions

Tell me about yourself. ▼
Formula: Present → Past → Future. "I'm a frontend developer specializing in [X]. I've built [specific project]. Before this, I [background]. I'm excited about this role because [specific reason]." Aim for 90 seconds maximum.
Tell me about a technical challenge you solved. ▼
Use STAR: Situation → Task → Action → Result. Be specific: "While building X, I hit Y bug. I debugged by [process], discovered Z was the cause, and fixed it by [solution]. Result: [outcome]." Specificity impresses more than generality.

Common Developer Mistakes
Learn from mistakes almost every developer makes. Recognizing them is half the battle.

Beginner Mistakes

  • Tutorial hell — consuming courses without building real projects
  • Trying to learn everything at once instead of going deep on one thing
  • Not using Git — "I'll add it later" becomes "I lost everything"
  • Skipping error messages and Googling before reading them
  • Copying code without understanding what it does
  • Hardcoding API keys into JavaScript files
  • Using <div> for buttons instead of <button>

Intermediate Mistakes

  • Over-engineering before validating the idea
  • Not handling fetch() errors — assuming it always succeeds
  • Mutating state directly in React instead of using setState
  • Never testing with slow connections or real mobile devices
  • Writing code only the author can understand
  • Not sanitizing user input — XSS and injection vulnerabilities
💡
When stuck: 1) Read the error. 2) console.log each step. 3) Search the exact error message. 4) Ask with a clear problem description. 5) Take a break — bugs get obvious after you step away.

Checklists
Launch Checklists
Everything to verify before pressing the launch button.

Website Launch Checklist

  • All pages load without errors
  • All links work — internal and external
  • Contact form sends and receives emails
  • Mobile responsive — tested on real device
  • Page speed acceptable (PageSpeed Insights)
  • Favicon is set
  • Meta title and description on every page
  • Open Graph image and tags set
  • SSL certificate active (HTTPS)
  • Analytics installed
  • Sitemap submitted to Google Search Console

First Client Checklist

  • Proposal sent and accepted in writing
  • 50% deposit received before starting
  • Timeline agreed — in writing
  • All required assets received from client
  • Client approved design before development started
  • Client approved final site before final payment
  • Final 50% payment received before file handover
  • Client credentials (hosting, domain) transferred
  • Testimonial or review requested

Freelancer Launch Checklist

  • Portfolio site live with 3+ projects
  • Services and pricing defined
  • Payment method ready (Paystack, bank account, etc.)
  • Proposal template ready to send
  • Social media presence active
  • First 3 potential clients identified and contacted
💡
The #1 freelance insight: Your first clients don't come from job boards. They come from people you already know. Tell everyone what you do and what problems you solve.
